# Memory Routing — 自动路由复习与复盘机制

_讨论时间：2026-05-10_

## 背景

当前记忆路由系统依赖**手动审计**：批量添加条目后，准确率从 95% 下降到 83%，需要人工跑全量审计 + 关键词调优循环（add-remove-add）才能恢复。问题：

1. **不会自动触发** — 需要星野主动想起去调
2. **错误即时发生** — 新增条目可能路由到错误文档，直到下次审计才发现
3. **基础设施黑洞** — infrastructure 关键词膨胀后"吸走"其他文档条目，失衡不会自动纠正
4. **维护负担** — 每次批量写入后的审计是全量的，耗时

## 现状痛点

| 问题 | 表现 | 频率 |
|------|------|------|
| 关键词需要人工调 | 写入时不会自动优化 | 每次批量添加后 |
| 低分路由即时错误 | score 0-2 的条目可能去错地方 | 每次写入 |
| 基础设施黑洞 | 40+ 关键词的子文档吸走条目 | 持续增长 |
| 全量审计慢 | 扫描所有条目（165+） | 每次都要从头跑 |

## 方向一：实时更新（写入时）

### 设计

```
memory(action="add", content="...")
  → route_content_to_sub_doc(content)
  → if score >= 3:
      → 写入子文档（原流程）
  → if score == 0-2:
      → 写入子文档 + 同时记录到 audit.log（不写入记忆）
      → 累计到 N 条低分条目后，触发轻量级复盘
```

### 复盘逻辑

1. 读取 audit.log 中的低分条目
2. 用 LLM（10.10.4.81:11434 Qwen3-4B）批量判断"应该去哪个文档"
3. 统计高频出现的分类，提取缺失关键词
4. 自动 patch 到 `memory_tool.py` 的 `SUB_DOCS`

### 好处

- 错误不会长期累积
- 每次写入都在"训练"路由系统
- 渐进式改进，不需要全量审计

### 风险

- LLM 分类可能不准（尤其对短文本）
- 自动 patch 关键词可能引入新问题（需要人工确认？）
- 审计日志如何管理（定期清理？）

## 方向二：定期复盘（定时）

### 设计

```
每 72 小时（结合 Curator 或 cron job）：
  → 增量审计（只审计最近 50 条条目，不是全量）
  → 计算增量准确率
  → if 准确率 < 80%:
      → 触发 add-remove-add 关键词调优循环
      → 自动 patch SUB_DOCS
      → importlib.reload() 应用
```

### 好处

- 增量审计比全量快得多
- 利用现有 Curator 基础设施
- 阈值触发，不需要人工判断

### 风险

- 72 小时间隔内可能有错误路由
- 增量审计的窗口大小需要调优（50 条？100 条？）

## 开放问题

1. **自动修改 vs 建议模式** — 棣民尚未决定：星野自动 patch 关键词，还是先提建议、棣民确认后改？
2. **复盘范围** — 只关注路由准确率，还是也关注记忆条目内容质量（过时信息、冗余合并）？
3. **审计日志存储** — 本地文件（~/.hermes/audit.log）还是存入 Qdrant？
4. **与现有 Curator 的关系** — 独立 cron job 还是合并到 Curator 的 interval？
